## 1. 개발 환경 구성

### 왜 환경 구성이 중요한가?
> "환경 구성은 서버와 같이 하는 게 좋다. 가장 동일한 형태, 가장 비슷한 환경으로 해놓고 작업해라."

실제 서버 환경(리눅스)과 내 로컬 환경이 다르면, 내 컴퓨터에서는 되는데 서버에서 안 되는 문제가 생긴다. 이걸 방지하기 위해 서버와 동일한 환경을 맞춰두는 것.

### 설치한 도구들

**Ubuntu (우분투)**
개발 환경과 운용 환경을 맞추기 위해 설치. 실제 서버 대부분이 리눅스(우분투) 기반이라 로컬도 맞춰두는 것.

**Git**
```bash
winget install Git.Git
```
형상관리(버전관리) 프로그램. 코드의 변경 이력을 기록하고 관리함.
- **Git**: 내 컴퓨터에서 버전을 관리하는 프로그램
- **GitHub**: 실질적인 외부 저장공간 (레포지토리)

**VS Code (Visual Studio Code)**
프로그램 기반의 에디터. AWS 스폰서와 직접적인 연결이 가능함.

**Terraform**
코드 기반으로 인프라를 생성하는 선언적 언어.
- 버전 업이 매우 빠르게 일어나는 편
- 명령어 구성이 완전히 바뀌는 때도 있어서 버전 확인이 중요함
- 프로젝트는 전부 코드 기반으로 사용, 콘솔은 10%만 사용

**kubectl**
Kubernetes를 사용할 때 쓰는 도구. 서버 수십 대를 한꺼번에 다룰 수 있음.

## 2. 클라우드 vs 온프레미스

| 구분 | 설명 |
|---|---|
| 온프레미스 | 물리적 구성으로 되어있는 것 (직접 서버 장비를 구매해서 운영) |
| 클라우드 | 논리적 구성으로 되어있는 것 (인터넷으로 빌려서 사용) |

### 클라우드의 장점
1. **자원을 아껴 쓸 수 있다** — 필요할 때만 쓰고 반납 가능
2. **어디서나 작업이 가능하다** — 인터넷만 있으면 접근 가능

## 3. 주요 개념 정리

### 이미지(Image)
두 가지 종류가 있음.
- **AMI**: AWS에서 제공하는 이미지 (EC2 인스턴스 만들 때 사용)
- **Docker 이미지**: 여러 개의 계층으로 이루어진 하나의 파일. 필요 없는 건 제거한 모습. 이 이미지를 만드는 게 Docker.

### 가용 영역 (Availability Zone)
- 최소 100km 서로 떨어져 있음
- 한 곳에 문제가 생기더라도 다른 곳은 사용 가능하게 함
- 기본적으로 세 개의 가용 영역을 가지고 있음

### EC2 관련 핵심 개념
- **EC2 인스턴스**: 실제 가상 서버
- **AMI**: 인스턴스를 만들기 위한 이미지 (설치된 OS, 소프트웨어 포함)
- **시작 템플릿(Launch Template)**: 인스턴스를 만들기 위한 설계도

### IAM 핵심
> "IAM에서 중요한 건 이 사람에게 어떤 역할을 줄 것인가 — 정책, 역할"

- **정책(Policy)**: 무엇을 할 수 있는지 정의
- **역할(Role)**: 정책을 묶어서 특정 서비스나 사람에게 부여

## 4. 유용한 명령어/개념 메모

### curl
```bash
curl [URL]
```
웹상에서 뭔가 다운받을 때 사용하는 명령어.

### 명령어 옵션 규칙
```
-  → 한 글자 옵션 (예: -o)
-- → 단어 옵션 (예: --output)
-o → 아웃풋(출력) 관련 옵션
```

### 쉘 스크립트(Shell Script)
```bash
sh [파일명]
```
`sh` 뒤에 나오는 파일을 실행하라는 뜻. 여러 명령어를 묶어서 자동으로 실행할 수 있게 만든 파일.

### 터미널 복붙 팁
코드 창에서는 `Ctrl+V`가 안 됨 → **마우스 오른쪽 클릭**으로 붙여넣기.
