## 📑 Core Essence: 개념별 매커니즘 심층 분석

### 1. AWS 가상 인프라 기초 공사와 원격 접속의 원리

#### 🖥️ AWS EC2 (Elastic Compute Cloud)의 본질
인터넷(클라우드) 너머 아마존 데이터 센터에 부팅되어 있는 고성능 가상 컴퓨터를 한 대 대여하는 서비스입니다. 
* **Elastic(탄력적인)**: 컴퓨터의 뇌(CPU)나 메모리(RAM), 하드디스크 용량을 마우스 클릭 몇 번으로 늘렸다가 줄일 수 있음을 뜻합니다.
* **Compute Cloud**: 하늘의 구름처럼 보이지 않는 거대한 서버 컴퓨터 무리에서 내 지분을 떼어 쓴다는 의미입니다. 24시간 내내 켜져 있으며 전 세계 누구나 접속할 수 있는 나만의 웹 서버를 구축하기 위한 물리적 기초 주춧돌이 됩니다.

#### 🔑 자물쇠와 마스터키: 비대칭 키페어(Asymmetric Key Pair) 매커니즘
AWS 컴퓨터는 전 세계 해커들의 무차별 대입 공격(Brute Force) 표적이 되기 때문에, 일반적인 ID/비밀번호 방식은 아예 문을 걸어 잠그고 사용할 수 없게 만듭니다. 대신 수학적으로 완벽하게 맞물린 한 쌍의 **비대칭 키페어** 방식을 사용합니다.

* **🔒 Public Key (공개키 자물쇠)**: AWS 데이터 센터에 있는 내 EC2 서버 내부에 설치되는 특수 자물쇠입니다. 이 자물쇠는 외부 세상에 노출되어도 안전합니다.
* **🔑 Private Key (개인키 마스터키 - `*.pem`)**: 가입할 때 단 한 번 내 노트북(로컬 환경)으로 다운로드해 주는 유일한 마스터키 원본입니다.
* **🔗 수학적 인증 결합**: 내가 접속을 시도하면, AWS의 자물쇠가 무작위 숫자로 된 암호화 퀴즈를 던집니다. 내 로컬 노트북에 있는 `*.pem` 마스터키가 이 퀴즈를 수학적으로 완벽하게 풀어내야만 서버의 문이 열립니다. 이 키 파일이 유실되거나 노출되면 서버의 제어권을 통째로 빼앗기므로 인프라 보안의 최우선 관리 대상입니다.

#### 🌐 퍼블릭 IP (Public IP)의 한계와 고정 IP(Elastic IP)의 당위성
가상 컴퓨터가 세상과 통신하기 위해 전 세계에서 유일하게 부여받은 인터넷 주소입니다. 하지만 EC2가 기본으로 받는 퍼블릭 IP에는 치명적인 약점이 있습니다. **"컴퓨터를 껐다 켜면(재부팅) 주소가 마음대로 바뀐다"**는 점입니다.
* 만약 홈페이지 주소가 매번 바뀐다면 손님들이 찾아올 수 없습니다. 
* 그래서 인프라 실무에서는 한 번 할당받으면 절대 바뀌지 않는 전화번호 같은 **고정 public IP(AWS 명칭: 탄력적 IP / Elastic IP)**를 서버에 반드시 바인딩(연결)해 주는 작업이 필수적으로 수반됩니다.

#### 🚀 SSH (Secure Shell) 보안 터널링의 원리
내 노트북(윈도우)에서 저 멀리 있는 리눅스 컴퓨터 내부로 들어가 명령어를 내리는 껍데기(Shell) 통로입니다. 과거의 원격 접속 도구(Telnet 등)는 내가 치는 명령어와 비밀번호가 인터넷 세상에 날것(평문)으로 흘러가 해커들이 중간에 훔쳐볼 수 있었습니다.
* **Shell**: 리눅스 커널(알맹이)을 감싸고 있는 껍데기로, 사용자의 명령어를 컴퓨터가 알아듣게 번역해 주는 인터페이스입니다.
* **Secure(보안)**: SSH는 내 노트북부터 AWS EC2 서버까지의 통신 경로를 100% 암호화된 가상의 파이프라인(보안 터널)으로 감쌉니다. 중간에 해커가 데이터를 가로채도 무작위 외계어로 보이기 때문에 완벽한 보안이 유지됩니다.
* 파워쉘을 켜고 `ssh -i "마스터키.pem" ec2-user@IP주소`를 치는 순간, 이 암호화 터널이 뚫리면서 내 노트북 키보드가 저 멀리 아마존 데이터 센터에 있는 리눅스 컴퓨터의 두뇌와 다이렉트로 동기화되는 매커니즘이 완성됩니다.

### 2. 소스 코드 배포(Deploy)와 경로 제어 시스템

#### ⚙️ 웹 서버 소스 코드 배포(Deploy)의 메커니즘
배포란 내 로컬 PC에서 제작하고 검증한 결과물을 원격(Remote) 라이브 서버로 안전하게 수송하여 인터넷 손님들이 접근할 수 있도록 가동하는 전체 과정을 뜻합니다.
1. **로컬 1차 확인**: 배포하기 전, 반드시 내 PC에서 코드의 정상 작동 여부를 선제적으로 확인하는 방어적 검증 단계를 거쳐야 합니다.
2. **배포 파이프라인**: 훗날 실무에서는 Git이나 Action 등을 활용한 자동화 파이프라인을 구축하게 되지만, 인프라의 본질과 에러를 완벽하게 제어하기 위해서는 **최초에 명령어를 수동으로 입력하여 배포하는 로직을 완벽히 이해**하고 있어야 합니다.

#### 🛸 파일 수송선 (`scp` - Secure Copy Protocol)
내 로컬 노트북(윈도우)에 저장된 웹사이트 구성 파일 3대장(`index.html`, `style.css`, `app.js`)을, SSH 보안 통로를 활용해 원격 리눅스 컴퓨터로 복사·전송하는 암호화 수송선 명령어입니다.
* **작동 원리**: 리눅스는 기본적으로 보안을 위해 외부인의 접근을 철저히 통제합니다. 따라서 AWS를 생성할 때 발급받은 마스터키 원본 파일(`-i` 옵션 뒤의 `*.pem` 파일)을 인증서로 제시해야만 수송선이 가상 컴퓨터 내부로 진입할 수 있습니다.
* **최종 수신지**: 기본적으로 외부에서 접속할 때는 보안상 안전한 유저의 개인 방(`ec2-user` 계정의 홈 디렉토리)을 수신 주소로 지정(`~ /`)하여 파일을 안전하게 이사시킵니다.

#### 🗺️ 절대경로 vs 상대경로의 명확한 차이
리눅스 파일 시스템에서 특정 방(디렉토리)이나 파일의 위치를 찾아가는 내비게이션 주소 지정 방식은 두 가지로 나뉩니다.
1. **절대경로**: 내 현재 위치가 어디든 상관없이, 언제나 시스템의 뿌리 기점인 최상위 총본부(어미 슬래시 `/`)에서부터 모든 방 이름을 순서대로 다 적어주는 **'불변의 주소'** 방식입니다. (예: `/var/www/html/index.html`). 중요한 시스템 파일이나 웹 서버의 절대적 근간 위치를 찾아갈 때 작업자의 혼선을 완벽히 방지해 줍니다.
2. **상대경로**: **"지금 내가 서 있는 방"**을 기준으로 길을 안내하는 유연한 방식입니다. 현재 방은 점 하나(`.`), 내 바로 위의 부모 방은 점 두 개(`..`)로 표시합니다. 내가 지금 `/var/www/` 방에 서 있다면, `html/index.html`이라고만 적어도 알아듣는 방식입니다.

#### 🌍 세계 공통 웹 진열대 (`/var/www/html/`)의 단어별 구조 분석
아파치 웹 서버가 외부 손님에게 웹페이지를 서빙하기 위해 무조건 바라보도록 뇌에 고정(기본값)되어 있는 약속된 표준 주소입니다. 이 긴 이름은 무작위로 지어진 것이 아니라 리눅스 표준 규격에 따른 명확한 유래가 있습니다.
* **`/` (루트)**: 리눅스 컴퓨터의 모든 세포가 시작되는 가장 첫 번째 최상위 본부 공간입니다.
* **`var` (Variable)**: 가동 중에 로그 기록이나 웹 파일처럼, 내용이나 용량이 자꾸자꾸 추가되고 '변할 수 있는(Variable) 데이터'들을 따로 격리해 놓은 전용 구역입니다.
* **`www` (World Wide Web)**: `var` 구역 안에서 특별히 '인터넷 웹 서비스 관련 프로그램'들만 모여서 활동하도록 지정된 방입니다.
* **`html`**: 진짜 웹사이트의 소스 코드 원본 파일(`index.html` 등)들이 최종적으로 안착하여 인터넷 손님들을 맞이하는 '진짜 진열대'입니다.
* **따로 격리하는 이유**: 내 개인 방 (`~`)에는 개인적인 연습 파일들이 잔뜩 섞여 있습니다. 만약 아파치 요원에게 내 개인 방을 뒤지게 하면 보안상으로도 위험하고 엉망진창이 되므로, "손님용 물건은 1층 로비 전용 진열대(`/var/www/html/`)에 딱 정리해 두자"고 분리해 놓은 것입니다.

### 3. 웹 서버 구동과 관리자 권한 통제

#### 🏢 웹 서버 엔진 선택의 매뉴얼화
웹 서비스를 띄우기 위한 엔진 소프트웨어를 선택할 때는 어떤 운영체제 위에 아키텍처를 세우는지 명확하게 매뉴얼화해야 합니다. 대표적으로 아마존 리눅스 계열에서는 `httpd`라는 이름으로 아파치 요원을 가동하며, 우분투 계열에서는 `apache2`라는 규격을 사용합니다. 고성능 리버스 프록시나 경량화가 필요할 때는 `Nginx` 등의 엔진을 선택지로 두고 인프라 목적에 따라 교체 사용합니다.

#### 🛡️ `sudo` 관리자 도장과 파일 이동 (`mv`) 프로세스
로컬에서 원격 컴퓨터의 독방(`~`)으로 쏘아 올린 소스 코드 파일들을 진짜 인터넷 손님 진열대인 `/var/www/html/`로 옮기려면 파일 이동 명령어인 `mv (Move)`를 사용해야 합니다.
* 이때 루트(`/`) 하위의 공용 진열대는 보안이 매우 삼엄하므로, 일반 계정 권한으로는 진입 및 파일 수정이 전면 차단됩니다.
* 따라서 명령어 맨 앞에 최고 최고 존엄 관리자의 승인 인장인 **`sudo` (SuperUser Do)** 명령어를 반드시 접두어로 붙여주어야만 시스템이 거부하지 않고 안전하게 등기 진열대로 파일을 이동시켜 줍니다.

#### 🔐 보안 자물쇠 비밀번호 (`chmod 644`)의 수학적 비밀
인터넷 손님이 웹브라우저 주소창에 내 IP를 치고 들어왔을 때, 컴퓨터가 "이 파일은 외부인이 볼 수 없는 보안 파일입니다"라며 `403 Forbidden` 에러를 뿜지 않게 하려면 파일 자물쇠의 설정을 변경해야 합니다. 리눅스는 권한을 **[읽기=4], [쓰기=2], [실행=1]**이라는 숫자의 합으로 계산합니다.

`chmod 644`라는 세 자릿수 숫자는 완벽하게 고정된 세 개의 계급 서열을 통제합니다.

1. **첫 번째 자리 [6] - 소유자(Owner) 권한**: 파일을 만든 주인(나)의 권한입니다. `4(읽기) + 2(쓰기) = 6`이므로, 나는 이 웹 소스코드를 언제든 열어보고 코드를 수정하여 저장할 수 있습니다.
2. **두 번째 자리 [4] - 그룹(Group) 권한**: 나와 같은 연구실이나 팀에 속한 팀원들의 권한입니다. `4(읽기)`만 부여되었으므로 팀원들은 코드를 열어볼 수는 있지만 마음대로 수정할 수는 없습니다.
3. **세 번째 자리 [4] - 기타 사용자(Others) 권한**: **전 세계 인터넷을 타고 들어오는 손님들이 바로 이 계급에 속합니다.** 이 자리에 최소 `4(읽기)` 권한이 있어야만 외부 웹브라우저가 이 파일을 읽어서 화면에 띄워줄 수 있습니다. 만약 이 숫자가 0이나 2라면 외부인은 절대 내 홈페이지를 볼 수 없습니다.

* **💡 권한 응용 오답 노트**:
  * **`664`**: 만약 협업하는 팀원들에게도 나와 동등하게 코드를 수정할 권리를 주고 싶다면, 두 번째 자리에 쓰기(`2`)를 더해 `664`로 세팅하는 것이 안전합니다.
  * **`614`**: 두 번째 자리에 실행(`1`) 권한만 덜렁 주는 설정입니다. 웹 페이지 파일(`html`)은 프로그램처럼 실행하는 파일이 아닌 텍스트 파일이므로, 읽지도 쓰지도 못하게 막고 실행만 하라는 설정은 리눅스 보안 관점에서 매우 모순적이고 기괴한 오답 세팅이 됩니다.

#### 🪪 명의 등기 이전 (`chown apache:apache`)의 당위성
수송선(`scp`)을 타고 처음 들어온 파일의 주인은 전송자인 `ec2-user` 계정으로 등록되어 있습니다. 하지만 실제 1층 로비에서 서빙을 담당하며 외부 손님에게 웹페이지를 실시간으로 전달하는 주체는 아파치 웹 서버 프로그램, 즉 `apache`라는 가상의 요원 계정입니다.
* **왜 명의를 바꿔야 하나요?**: 주인이 `ec2-user`로 되어 있으면, `apache` 요원이 손님이 올 때마다 남의 방에 있는 파일을 눈치 보며 꺼내와야 하므로 보안 제약이 걸리거나 속도가 지연될 수 있습니다. 
* **구조적 해결**: `chown apache:apache` 명령을 통해 파일의 앞 주인(User)과 뒤 그룹(Group)의 명의를 전부 아파치 요원 앞으로 완전히 등기 이전해 줍니다. 이로써 아파치 요원은 완벽한 정당성과 소유권을 가지고 외부 손님에게 가장 빠르고 안전하게 웹 사이트 서비스를 제공할 수 있게 됩니다.

### 4. 인터페이스 시스템과 OS별 홈 주소 심볼 체계

#### 🗺️ 운영체제별 경로 표시자(Slash)의 본질적 방향 차임
컴퓨터가 파일과 폴더의 계층을 타고 내려갈 때 사용하는 구분 기호는 OS 가문의 역사에 따라 완전히 정반대로 찢어집니다.
* **리눅스 / 우분투 계열**: 오른쪽으로 쓰러지는 모양의 **슬래시 (`/`)**를 표준 주소 구분 기호로 채택합니다.
* **윈도우 계열**: 왼쪽으로 쓰러지는 모양의 **역슬래시 (`\`)**를 주소 구분 기호로 사용합니다. (한국어 윈도우 글꼴 환경에서는 원화 기호 `₩`로 변환되어 표기됩니다.)

#### 🏠 로그인한 내 계정의 안전한 개인방: '홈 디렉토리' 3대 가문별 심볼 분석
운영체제와 가동하는 명령어 창(Shell) 엔진의 종류에 따라, 내 개인 독방 주소를 호출하고 대변하는 고유 신분증 문법(Symbol)이 엄격하게 분리됩니다.


1. **리눅스 가문 (Linux / Ubuntu)**
   * **물리적 절대경로**: `/home/사용자명` (예: `/home/ec2-user`)
   * **대변 심볼**: **`~` (물결 기호)**. 명령어 창에 `cd ~`라고 치면 시스템의 복잡한 경로를 생략하고 즉시 내 독방으로 순간이동합니다.
2. **윈도우 구형 가문 (명령 프롬프트 - CMD)**
   * **물리적 절대경로**: `C:\Users\사용자명`
   * **대변 심볼**: **`%USERPROFILE%`**. 퍼센트 기호로 양 끝을 감싸 구형 윈도우 환경 변수 주머니에서 경로를 호출하는 유산 문법입니다.
3. **윈도우 신형 가문 (파워쉘 - PowerShell)**
   * **물리적 절대경로**: `C:\Users\사용자명`
   * **대변 심볼**: **`$env:USERPROFILE`**. 파워쉘의 강력한 마법 기호인 달러(`$`)와 환경 구역 지시자(`env:`), 그리고 변수명을 콜론 기호로 정교하게 결합한 현대적 정석 문법입니다.

### 5. 클라우드 가상 네트워크의 주춧돌, 사설 IP와 보안 통신 체계

#### 🔒 사설 IP (Private IP)의 필요성
모든 컴퓨터가 외부 세상에 날것 그대로 노출되는 퍼블릭 IP만 사용한다면 해커들의 공격에 무방비로 노출됩니다. 따라서 클라우드 내부에 우리 팀만 사용하는 안전한 '가상 성벽(VPC)'을 쌓고, 그 안에서 컴퓨터끼리만 속닥속닥 대화할 수 있도록 내부용 주소를 부여하는데, 이를 **사설 IP**라고 합니다.

#### 🗺️ 약속된 3가지 사설 IP 주소 범위 (메뉴판)
우리가 만드는 네트워크의 규모(방이 몇 개나 필요한지)에 따라 골라 쓸 수 있도록 국제 규격에 의해 대·중·소 3가지 클래스로 나누어 제공됩니다.

| 규모 | 사설 IP 주소 시작 ~ 끝 범위 | 대표적인 표기법 (CIDR) | 주로 어디서 쓰나요? |
| :---: | :--- | :--- | :--- |
| **대형 (A클래스)** | `10.0.0.0` ~ `10.255.255.255` | `10.0.0.0/8` | **AWS VPC 구축할 때 가장 많이 쓰는 표준 범위** (대기업 내부망) |
| **중형 (B클래스)** | `172.16.0.0` ~ `172.31.255.255` | `172.16.0.0/12` | 학교나 일반 기업 사내 네트워크 |
| **소형 (C클래스)** | `192.168.0.0` ~ `192.168.255.255` | `192.168.0.0/16` | 우리 집 공유기, 카페 와이파이 내부망 |

#### 🚪 사설망의 일방통행 비밀 출입구, NAT 게이트웨이(Network Address Translation)
외부 해커의 침입을 원천 차단하기 위해 인터넷이 아예 연결되지 않는 깊숙한 성벽 내부(Private Subnet)에 둔 컴퓨터들은 안전하지만, **"리눅스 시스템 업데이트를 하거나 외부 라이브러리를 다운로드하는 바깥으로의 통신"**까지 전부 막혀버리는 모순이 발생합니다. 이 문제를 해결하는 구원투수가 바로 NAT 게이트웨이입니다.


* **주소 변환(Translation) 메커니즘**: 격리된 내부 컴퓨터가 외부 구글 서버로 데이터를 요청하면, NAT 게이트웨이가 중간에 패킷을 가로채서 **"컴퓨터의 사설 IP(Private IP) 신분증은 숨겨두고, 내가 가진 공인 IP(Public IP) 신분증으로 교체"**해서 바깥세상으로 보내줍니다.
* **완벽한 보안 유지 원리**: 내부에서 밖으로 나가는 요청만 주소를 바꾸어 내보내 주기 때문에, **"안에서 밖으로 나가는 통신"은 자유롭지만, 반대로 "외부 해커가 밖에서 안으로 직접 쳐들어오는 통신"은 문 자체가 존재하지 않아 완벽하게 차단**됩니다.

#### ⚖️ 인터넷 게이트웨이(IGW) vs NAT 게이트웨이 구조적 차이
클라우드 인프라 아키텍처에서 외부와 통신하는 문은 두 종류로 엄격하게 구별하여 배치합니다.

1. **인터넷 게이트웨이 (Internet Gateway - IGW)**: **[양방향 광장]** 외부 손님들이 주소창을 치고 내 홈페이지에 들어올 수도 있고(`Inbound`), 내 서버가 밖으로 나갈 수도 있는(`Outbound`) 완전히 열린 통로입니다. 웹 서버가 있는 퍼블릭 구역(Public Subnet)에 연결합니다.
2. **NAT 게이트웨이 (NAT Gateway)**: **[일방통행 보안문]** 내부 컴퓨터들이 오직 업데이트 등을 위해 밖으로 나갈 수만 있고(`Outbound` 전용), 외부에서는 이 문을 타고 역으로 절대 들어올 수 없습니다. 데이터베이스(DB)나 핵심 백엔드 서버가 숨어있는 프라이빗 구역(Private Subnet)의 컴퓨터들을 위해 배치합니다.

## ⚡ Quick Cheat Sheet: 인프라 구축 및 실전 배포 명령어 요약

가상 컴퓨터 접속부터 로컬 웹 파일을 AWS EC2(아마존 리눅스)에 올리고 최종 가동할 때까지 사용하는 전체 명령어 프로세스입니다.

| 순서 | 명령어 실행 환경 | 실행 명령어 (Command) | 목 적 (Purpose) |
| :---: | :---: | :--- | :--- |
| **1** | 로컬 파워쉘 | `ssh -i "[키페어.pem]" ec2-user@[공인IP]` | AWS 원격 가상 컴퓨터 내부로 보안 접속 |
| **2** | 로컬 파워쉘 | `scp -i [키페어.pem] [파일명] ec2-user@[공인IP]:~/` | 로컬 웹 파일을 AWS 개인 독방(`~`)으로 전송 |
| **3** | 원격 리눅스 | `sudo dnf -y install httpd` | 아파치 웹 서버 가이드 요원 엔진 설치 |
| **4** | 원격 리눅스 | `sudo mv ~/[파일명] /var/www/html/` | 웹 파일을 공식 인터넷 손님 진열대로 이동 |
| **5** | 원격 리눅스 | `sudo chmod 644 /var/www/html/[파일명]` | 인터넷 손님이 읽을 수 있게 파일 자물쇠 해제 |
| **6** | 원격 리눅스 | `sudo chown apache:apache /var/www/html/[파일명]` | 파일 소유권을 아파치 요원에게 등기 이전 |
| **7** | 원격 리눅스 | `sudo systemctl start httpd` | 아파치 웹 서버 가동 및 실시간 배포 시작 |

## 🚨 Troubleshooting & Trial Error (시행착오 기록)

### 01. 파워쉘 환경 변수 호출 문법 오류
* **현상**: 파워쉘 환경에서 사용자 프로필 경로를 호출할 때 구분자 오류 발생.
* **원인**: 옛날 윈도우(CMD) 방식인 `%USERPROFILE%`이나 세미콜론(`;`)을 사용하면 파워쉘 내부 엔진이 정확히 인식하지 못함.
* **해결**: 파워쉘 정석 문법인 콜론(`:`)을 활용해 **`$env:USERPROFILE`**로 호출해야 안전하고 정확하게 작동함.

### 02. `chmod` 보안 권한 조합의 미스매치
* **현상**: 파일 권한을 `614` 등으로 설정 시 아파치 웹 서버 서빙 로직에서 어색함 발생.
* **원인**: 두 번째 자리(그룹/팀원)에 실행(`1`) 권한만 주고 읽기(`4`)를 막는 것은 소스 코드 파일 관리 관점에서 매우 이례적이고 위험한 설정임.
* **해결**: 외부 손님이 읽어야 하는 웹 소스는 **`644`**(기본)로 세팅하거나, 팀원에게도 수정 권한을 주려면 쓰기(`2`)를 더한 **`664`**를 채택하는 것이 정석임.
